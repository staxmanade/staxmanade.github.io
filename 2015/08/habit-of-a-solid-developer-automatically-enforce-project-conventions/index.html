Habit of a Solid Developer - Part 2 - Automatically Enforce Project Conventions

Aug 24th 2015 (Comments)

Every project generates it's own constraints. These may come from an architects's idea of "best practice" or it may come from the domain the software is written in, it may even be derived because of the language you're using. For example, in JavaScript we have linters that enforce rules like triple equals vs double equals.

The class of convention can be wide raging, anything from the style or formatting of the code itself, or naming conventions of objects, to obscure ways of doing X with component Y that if not done in that way will cause problems.

Automating these convention will help to not only keep the project on track if a newer developer is onboard and isn't aware of the convention, but can even catch the most senior developer when they forget the convention all together.

Example of a project convention

In the past I worked on a mutual fund and stock trading engine. This engine depended heavily on time in places and deep down in the core of the application we built in an abstraction over the core Date implementation. This abstraction allowed us to do things like roll the date back for testing or debugging purposes. But if someone ever used a new Date() in the code, this could potentially break this convention and not allow us to roll time when we required it.

The above can be caught by running code reviews, but what if you had a test failure raise the issue before the code was even committed?

This is a perfect scenario for automation. We simply put a unit test that searched through the code base for uses of new Date() and the test would fail if it found any of them.

Fail with solid path forward

The test described above would not just fail with a generic do not use new Date()... but is a good place to put a story about why new Date() was not allowed. So we would instruct the developer in the error message both why not to use new Date() but what to use in place of. You want to include the why here to easily educate those who are not aware of this convention or rule.

Provide a way to add exclusions or exceptions to the convention

For all conventions, these are generalizations that may not ALWAYS be necessary or ideal. There are almost always exceptions to the rule in software development, so allowing for the test code to be modified to enable exclusions to this rule is a good thing.

Some Automation Approaches

Depending on the convention, the approach to automation can vary. Sometimes it's as easy as adding a simple unit test. Other times it requires you jump out to a command line utility that searches the entire codebase. Whatever the convention, try to come up with a simple approach that can automatically be run through a continuous integration environment and can freely be there to watch over you and your team's conventions.
