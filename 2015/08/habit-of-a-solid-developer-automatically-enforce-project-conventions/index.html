Habit of a Solid Developer - Part 2 - Automatically Enforce Project Conventions

Aug 24th 2015 (Comments)

Every project generates it's own constraints. These may come from an Architects idea of "best practice" or it may come from the domain the software is written in. It may even be derived because of the programming language you're using. For example, in JavaScript we have linters that encourage rules like triple equals === vs double equals ==.

The class of conventions can encompass a wide range, anything from the style or formatting of the code itself, or naming conventions of objects, to obscure ways of doing X with component Y that if not done in that way will cause problems.

Finding a way to automatically check and enforce conventions will help to not only keep the project on track if a newer developer is onboard and isn't aware of the convention to even catching the most senior developer when they forget the convention all together (It's happened to me - forget).

One example of a project convention

In the past, I worked on a mutual fund and stock trading engine. This entire system depended heavily on time in places and deep down in the core of the application we built in an abstraction over the core Date implementation. This abstraction allowed us to do things like roll the date back for testing or debugging purposes. Simulate billing over time rages. However, if someone ever used a new Date() in the code, this would break the convention and not allow us to roll time when we required it or even introduce hard to find bugs.

This is a great place for automation. The above can be caught by running code reviews. However, if this rule was baked into the system and automated the issue can be caught much earlier in the process. With a well-placed a unit/integration or build test that searched through the code base for uses of new Date() and raises an error message explaining why...

Fail with solid path forward

In our automated testing system, the convention/rule described above would not just fail a test with a generic do not use new Date()... message. This is a good place to document or share a story about why new Date() was not allowed. We failed the test with a long error message that described both why not to use new Date() but what to use in place.

You want to include the why as well as the what here to make adoption of the conventions easier going forward for those not aware of them.

Allow Special Exclusions

For most conventions, they are generalizations that may not ALWAYS be necessary or ideal. There often times exceptions to the rule in software development, so allowing the test code to be modified to enable exclusions to this rule is a good thing.

In the example convention of Date above we had a place in the unit test that allowed us to say "In component M method Y we allowed the use of a single case of new Date()". The first exercise of the exclusion was built in to the original abstraction over time. But there were a couple other places we allowed it as I recall.

Some Automation Approaches

Depending on the convention, the approach to automation can vary. Sometimes it's as easy as adding a simple unit test. Other times it requires you jump out to a command line utility that searches (greps) the entire codebase. Whatever the convention, try to come up with a simple approach that can automatically be run through a continuous integration environment and can freely be there to watch over you and your team's conventions.

Examples of tools to help enforce conventions

  • .editorconfig
  • ReSharper
  • Unit Test Projects
  • Build Scripts
  • Continuous Integration Environments
  • Pull Requests and Code Reviews
  • Gated check-ins
  • This could become a really large list...
