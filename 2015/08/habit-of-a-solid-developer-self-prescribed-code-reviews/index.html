Habit of a Solid Developer - Part 3 - Self Prescribed Code Reviews

Aug 24th 2015 (Comments)

One technique I use pretty regularly when working with a team or even by myself is to review code changes.

Depending on the project, how many developers, time zones of committers, etc I shift how often or how thorough I review the changes but I almost always review every single commit.

One good example on how I accomplished this was at a previous employer. I started work several hours before everyone else, and left before everyone. This meant that there was often end of day commits by other developers to review, and enough quiet time in the morning for me to take the time to do a solid review of the previous day's changes.

While reviewing the previous day's changes, I would not only review other's changes, but I would also review my own commits. This helped remind me what I worked on and possibly what I was struggling through the previous day. Doing this the next day provided enough time for my mental perspective to have shifted. I was probably thinking about things a little more clearly. Maybe I'd learned something that could be applied to the problem being worked on and provide better setup for changes needing to be made today.

This morning review process was good from a number of other angles:

  • It was a bit of a meditative process in the early morning that put my brain on a code thinking track for the day.
  • I could jot down questions on changes I couldn't understand (from others) which either led to asking for clarifications or even turned into possible suggestions that other developers could use to re-working their implementation even if it was as simple as (this part is unclear, could you add some extra comments here to help with clarity, or rename a variable to be a little more intention revealing).
  • Easily catch when the project conventions were not being followed.
  • Spot bugs being introduced.
  • Suggest potentially better approaches.

However, the most important thing I gained by doing this was:

  • Gain a better understanding of the entire project.
  • Learn from others.

I once had a fellow developer say to me that they were blown away at how quick I could spin up on a brownfield project and get an architectural understanding so quickly that I could start not only contributing, but spot the places where we could improve general architectural patterns or jump in and become productive on a problem set.

I believe that reviewing the changes going into a code base each day allowed me to keep a strong perspective on what is shifting within the project and be able to talk intelligibly when others start to get stuck on their work.

Code Reviews

If your team practices some form of code reviews, this is great for those participating in the review process. However it leaves the rest of the developers out of that specific review. However, this should not stop those not in the review room from participating and learning from the changes.

Many of the places I've worked don't have a formal review process in place, so by putting this little ritual in place for myself I can not only come up to speed on a project but can be helpful to all aspects of the project. Reviewing changes every day allows me to more easily understand what people were working on when they announce it during the morning standup.

Pull Requests

The example I gave above of reviewing commits that happened in the past did leave room for improvement. If we only had pull requests back then... If you're using GitHub or another system that supports a way to code review before the code is placed on the main line of code, this is obviously better than what I described above.

Just Review Everything

If you project is small enough and you can - spend a moment each day and review everything going on in the project. You and you're team will likely be better off.
