Habit of a Solid Developer - Part 5 - Document, While Reading Code

Dec 13th 2014 (Comments)

*** you're should be your *** While I generally sit on the side of the fence that you're code should speak enough that comments should not be necessary, I do agree that there are times when a comment here and there can really help to make the job of understanding the code much easier than just how a variable or method is named or organized.

*** it's should be its *** One example where I think a comment is due is when reading code and not really understanding what's going on. Some may say that a comment should have made it's way there initially, but those obvious to the original coder scenarios are not always as easy to know up front. So if you catch yourself digging through some code and a bit confused in an area, take the time to document it (once you figure out what you didn't understand).

Being stumped while reading code is a perfect opportunity and trigger that idea of adding a little clarity. (Or fixing a variable or method name).

Now go read these comments and enjoy!
